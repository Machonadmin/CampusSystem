# Модуль «Управление учёбой» (Study Management)

Этот документ описывает управление учёбой в разрезе учебной группы
(`class_groups`). Реализовано три составляющие:

1. **Уроки** (`lessons`) — журнал занятий группы: дата/время/тема/место.
2. **Посещаемость** (`attendance`) — отметка присутствия студентов на уроке.
3. **Оценки** (`assessments` + `grades`) — задания/работы группы и оценки
   студентов за них.

Все три реализованы как вкладки одной карточки учебной группы
(`ClassGroupCardClient.tsx`) и используют общий способ вычисления прав —
`PrivilegeTarget` группы (`department_id` + `teacher_ids` из
`class_teachers`), см. `lib/education/lesson-access.ts`. Расписание/семестры
— отдельный, ещё не реализованный этап.

Поверх этих трёх составляющих есть разрез **по студенту** — вкладка
«Успеваемость» карточки студента: сводка посещаемости и оценок по всем
группам, куда он записан. Это только чтение и агрегация (своих таблиц и
миграций нет); описана отдельным разделом «Отчёт об успеваемости студента»
ниже.

## Модель данных

`lessons`/`attendance` созданы миграцией `20260705150000_lessons_attendance.sql`,
`assessments`/`grades` — миграцией `20260705170000_grades.sql`. Обе миграции
**сами определяют** функцию `set_updated_at()` (`CREATE OR REPLACE FUNCTION
... RETURNS TRIGGER`) — на случай, если этой функции ещё нет в целевой БД;
это делает их самодостаточными и не зависящими от того, какие из более
ранних миграций (`20260608190000_add_updated_at_remaining.sql` и др.) были
применены.

### `lessons` (уроки учебной группы)

| Поле             | Тип         | Комментарий                                   |
|------------------|-------------|-----------------------------------------------|
| `id`             | uuid PK     | —                                              |
| `class_group_id` | uuid FK     | → `class_groups(id)`, `ON DELETE CASCADE`      |
| `scheduled_date` | date        | обязательно                                    |
| `scheduled_time` | time        | опционально                                    |
| `topic`          | text        | опционально, заполняется позже                |
| `description`    | text        | опционально                                    |
| `location`       | text        | опционально (аудитория/здание)                 |
| `is_cancelled`   | boolean     | `NOT NULL DEFAULT FALSE`                       |
| `created_by`     | uuid FK     | → `persons(id) ON DELETE SET NULL`             |
| `created_at` / `updated_at` | timestamptz | стандартные, `updated_at` — через триггер |

Уникальность: `UNIQUE (class_group_id, scheduled_date, scheduled_time)` —
в одной группе не может быть двух уроков на одинаковые дату+время.
Индекс `idx_lessons_class_group_date` на `(class_group_id, scheduled_date
DESC)` — под типовую выборку «уроки группы по убыванию даты».

### `attendance` (посещаемость, без оценок на этом этапе)

| Поле         | Тип       | Комментарий                                          |
|--------------|-----------|-------------------------------------------------------|
| `id`         | uuid PK   | —                                                      |
| `lesson_id`  | uuid FK   | → `lessons(id) ON DELETE CASCADE`                      |
| `journey_id` | uuid FK   | → `education_journeys(id) ON DELETE CASCADE`           |
| `status`     | text      | `CHECK (status IN ('present','absent','excused','late'))`, `NOT NULL`, без default |
| `marked_by`  | uuid FK   | → `persons(id) ON DELETE SET NULL`                     |
| `marked_at`  | timestamptz | когда была сделана отметка (не имеет default — заполняется приложением) |
| `created_at` / `updated_at` | timestamptz | стандартные, `updated_at` — через триггер |

Уникальность: `UNIQUE (lesson_id, journey_id)` — одна запись посещаемости
на пару «урок × студент»; повторная отметка обновляет её (см. API ниже).
Индексы `idx_attendance_lesson` и `idx_attendance_journey` — по каждому из
FK отдельно.

Статусов посещаемости четыре: **present** (присутствовал), **absent**
(отсутствовал), **excused** (уважительная причина), **late** (опоздал).
Отдельного значения «не отмечен» в БД нет — оно возникает на уровне API
как отсутствие строки `attendance` для пары (урок, студент); см. `GET
.../attendance` ниже.

### `assessments` (задания/работы учебной группы)

| Поле              | Тип          | Комментарий                                      |
|-------------------|--------------|----------------------------------------------------|
| `id`              | uuid PK      | —                                                   |
| `class_group_id`  | uuid FK      | → `class_groups(id)`, `ON DELETE CASCADE`           |
| `title`           | text         | обязательно                                         |
| `max_score`       | numeric(6,2) | `NOT NULL DEFAULT 100`, `CHECK (max_score > 0)`     |
| `assessment_date` | date         | опционально                                         |
| `description`     | text         | опционально                                         |
| `created_by`      | uuid FK      | → `persons(id) ON DELETE SET NULL`                  |
| `created_at` / `updated_at` | timestamptz | стандартные, `updated_at` — через триггер    |

Уникальных ограничений нет — в одной группе может быть несколько заданий
с одинаковым названием и/или датой. Индекс `idx_assessments_class_group_date`
на `(class_group_id, assessment_date DESC NULLS LAST)` — под выборку
«задания группы по убыванию даты, недатированные — последними».

### `grades` (оценки студентов за задания)

| Поле            | Тип          | Комментарий                                            |
|-----------------|--------------|----------------------------------------------------------|
| `id`            | uuid PK      | —                                                         |
| `assessment_id` | uuid FK      | → `assessments(id) ON DELETE CASCADE`                     |
| `journey_id`    | uuid FK      | → `education_journeys(id) ON DELETE CASCADE`              |
| `score`         | numeric(6,2) | `NOT NULL`, `CHECK (score >= 0)`                          |
| `comment`       | text         | опционально                                               |
| `graded_by`     | uuid FK      | → `persons(id) ON DELETE SET NULL`                        |
| `graded_at`     | timestamptz  | когда выставлена оценка (без default — заполняется приложением) |
| `created_at` / `updated_at` | timestamptz | стандартные, `updated_at` — через триггер        |

Уникальность: `UNIQUE (assessment_id, journey_id)` — одна оценка на пару
«задание × студент»; повторное выставление обновляет её (upsert, см. API
ниже). Индексы `idx_grades_assessment` и `idx_grades_journey` — по
каждому из FK отдельно.

Верхняя граница `score ≤ max_score` **не выражена как `CHECK` в БД** — это
ограничение между двумя таблицами, обычный `CHECK` его не выразит. Оно
реализовано на уровне API (см. «Две проверки уровня API» ниже); в БД
гарантируется только `score >= 0`. Как и с посещаемостью, отдельного
значения «оценка ещё не выставлена» в БД нет — оно возникает на уровне
API как отсутствие строки `grades` для пары (задание, студент); см. `GET
.../grades` ниже.

Сознательно отложено на будущее (см. комментарий в начале миграции
`20260705170000_grades.sql`): вес задания (`weight`) для взвешенного
итога, тип задания (экзамен / домашняя работа / проект), вычисляемая
итоговая оценка.

## API

Роуты уроков/посещаемости и роуты заданий/оценок используют общий хелпер
`lib/education/lesson-access.ts`, который строит `PrivilegeTarget` учебной
группы (`{ department_id, teacher_ids }`) для проверки прав —
`department_id` берётся из `class_groups.department_id`, `teacher_ids` —
список `teacher_id` из `class_teachers` этой группы. Это тот же способ,
каким уже вычисляют target существующие роуты `class-groups/[id]/*`.

- `getClassGroupTarget(sb, classGroupId)` — строит target по id группы.
  Используется в `class-groups/[id]/lessons` и `class-groups/[id]/assessments`
  (список/создание урока или задания).
- `getLessonAccess(sb, lessonId)` — загружает урок, затем строит target
  его группы через `getClassGroupTarget`. Используется во всех роутах,
  которые оперируют одним уроком (`lessons/[lessonId]` и его
  `attendance`).
- `getAssessmentAccess(sb, assessmentId)` — полная аналогия
  `getLessonAccess`: загружает задание (`assessment`), затем строит
  target его группы через `getClassGroupTarget`. Используется во всех
  роутах, которые оперируют одним заданием (`assessments/[id]` и его
  `grades`).
- `getEnrolledJourneyIds(sb, classGroupId)` — множество `journey_id`,
  записанных в группу; используется для счётчиков `enrolled_count`, для
  проверки, что отмечаемый студент действительно записан в группу урока
  (посещаемость), и для той же проверки при выставлении оценок.

### Уроки и посещаемость

| Метод + путь | Право | Описание |
|---|---|---|
| `GET /api/education/class-groups/[id]/lessons` | `view_students` (target группы) | Список уроков группы, по убыванию `scheduled_date`/`scheduled_time`. Каждый урок дополнен `marked_count` (сколько записей `attendance` у него есть); ответ также содержит `enrolled_count` (сколько студентов записано в группу) — фронт показывает это как «X / Y» в колонке посещаемости. |
| `POST /api/education/class-groups/[id]/lessons` | `set_lesson_topics` (target группы) | Создание урока. Тело: `scheduled_date` обязателен, `scheduled_time`/`topic`/`description`/`location` опциональны. `created_by` = `session.person_id`. |
| `GET /api/education/lessons/[lessonId]` | `view_students` (target группы урока) | Один урок + вся его посещаемость (`id, journey_id, status, marked_by, marked_at`), без наложения на список студентов группы. |
| `PATCH /api/education/lessons/[lessonId]` | `set_lesson_topics` (target группы урока) | Правка `scheduled_date`, `scheduled_time`, `topic`, `description`, `location`, `is_cancelled` (частичный апдейт — только переданные поля). |
| `DELETE /api/education/lessons/[lessonId]` | `set_lesson_topics` (target группы урока) | Удаление урока; посещаемость удаляется каскадно (`ON DELETE CASCADE`). |
| `GET /api/education/lessons/[lessonId]/attendance` | `view_students` (target группы урока) | Посещаемость, **наложенная на список записанных студентов**: каждый `journey`, состоящий в `class_enrollments` группы, возвращается со своим `status`/`marked_by`/`marked_at`, либо с `status: null`, если запись ещё не создана — это и есть UI-состояние «не отмечен». |
| `POST /api/education/lessons/[lessonId]/attendance` | `mark_attendance` (target группы урока) | Массовая отметка. Тело: `{ entries: { journey_id, status }[] }`. Перед сохранением каждый `journey_id` проверяется на членство в `class_enrollments` группы (иначе 400). Upsert по `(lesson_id, journey_id)` — `onConflict: 'lesson_id,journey_id'`; `marked_by` = текущий пользователь, `marked_at` = время запроса. |

Обработка ошибок — `mapDbError` в каждом файле (тот же паттерн, что в
`class-groups/route.ts`): `22P02` → 400 «неверный идентификатор», `23503`
→ 400 «ссылка на несуществующую запись», `23505` → 409 «урок на эту дату
и время уже существует» (только в роутах `lessons`), `23514` → 400
«недопустимый статус» (только в роуте `attendance`, хотя на практике этот
код недостижим — статус уже валидируется вручную до запроса к БД). Статус
`entries[].status` проверяется на принадлежность `present`/`absent`/
`excused`/`late` до обращения к БД — невалидное значение возвращает 400 с
перечислением допустимых.

### Задания и оценки

| Метод + путь | Право | Описание |
|---|---|---|
| `GET /api/education/class-groups/[id]/assessments` | `view_students` (target группы) | Список заданий группы, по убыванию `assessment_date` (`NULLS LAST`), вторичная сортировка по `created_at` по убыванию. Каждое задание дополнено `graded_count` (сколько строк `grades` у него есть); ответ также содержит `enrolled_count` (сколько студентов записано в группу). |

| `GET /api/education/assessments/[id]` | `view_students` (target группы задания) | Одно задание + все его оценки (`id, journey_id, score, comment, graded_by, graded_at`), без наложения на список студентов группы — аналогично `GET /lessons/[lessonId]`. |
| `PATCH /api/education/assessments/[id]` | `set_grades` (target группы задания) | Правка `title`, `max_score`, `assessment_date`, `description` (частичный апдейт). Если меняется `max_score` — см. вторую проверку в разделе ниже. |
| `DELETE /api/education/assessments/[id]` | `set_grades` (target группы задания) | Удаление задания; оценки удаляются каскадно (`ON DELETE CASCADE`). |
| `GET /api/education/assessments/[id]/grades` | `view_students` (target группы задания) | Оценки, **наложенные на список записанных студентов**: каждый `journey` из `class_enrollments` группы возвращается со своим `score`/`comment`/`graded_by`/`graded_at`, либо с `score: null`, если оценка ещё не выставлена. Ответ также содержит `max_score` задания (нужен фронту для валидации и подписи «из X»). |
| `POST /api/education/assessments/[id]/grades` | `set_grades` (target группы задания) | Массовое выставление оценок. Тело: `{ entries: { journey_id, score, comment? }[] }`. Порядок валидации — см. ниже. Upsert по `(assessment_id, journey_id)` — `onConflict: 'assessment_id,journey_id'`; `graded_by` = текущий пользователь, `graded_at` = время запроса. |

Обработка ошибок — `mapDbError` в каждом файле, тот же паттерн: `22P02` →
400 «неверный идентификатор», `23503` → 400 «ссылка на несуществующую
запись», `23514` → 400 («проверьте max_score» в роутах `assessments`,
«недопустимая оценка» в роуте `grades`; на практике недостижимо, так как
`score >= 0` и `max_score > 0` уже проверяются вручную до запроса к БД).

#### Две проверки уровня API (не `CHECK` в БД)

- **`score ≤ max_score` при выставлении оценки.** В БД `CHECK` есть только
  на `score >= 0` (см. модель данных выше) — верхняя граница не может быть
  выражена как `CHECK`, так как `max_score` лежит в другой таблице
  (`assessments`). Проверка — в `POST assessments/[id]/grades`: после
  получения `access.assessment.max_score` каждый `entries[].score`
  сравнивается с ним; при превышении — 400 с текстом вида `score 150
  превышает максимум 100 (journey …)`, где указаны и переданное значение,
  и лимит, и конкретный `journey_id`.
- **Нельзя понизить `max_score` ниже существующих оценок.** В `PATCH
  assessments/[id]`, если тело содержит `max_score`, перед обновлением
  выполняется `SELECT count(*) FROM grades WHERE assessment_id = … AND
  score > <новый max_score>` (через `count: 'exact', head: true`); если
  найдена хотя бы одна такая оценка — 400 с текстом вида `Нельзя
  установить max_score=50: 2 оценок превышают это значение. Сначала
  исправьте их.`, и обновление не выполняется. Это защищает от
  рассинхронизации, при которой в БД остались бы `grades.score >
  assessments.max_score`.

## Отчёт об успеваемости студента

В отличие от трёх вкладок карточки группы (разрез по `class_groups`),
отчёт об успеваемости — это разрез **по студенту** (`journey`): сводка
посещаемости и оценок по всем учебным группам, куда он записан. Только
чтение и агрегация уже описанных `lessons`/`attendance`/`assessments`/
`grades` — новых таблиц и миграций нет.

### Эндпоинт

`GET /api/education/journeys/[id]/report` — право `view_students`.
Возвращает `{ journey_id, summary, groups }`:

- `groups[]` — по одной записи на каждую **видимую** группу студента:
  `subject` (предмет), `name`/`level` группы, блок `attendance`
  (`present`/`absent`/`excused`/`late`, `marked`, `total_lessons`,
  `percent`) и блок `grades` (`graded_count`, `total_assessments`,
  `average`, `assessments[]` — все задания группы с оценкой студента
  или `null`).
- `summary` — свёртка по всем видимым группам: `visible_group_count`,
  суммарные блоки `attendance` и `grades`.

404, если `journey` не найден; `mapDbError` — тот же паттерн (`22P02` →
400, `23503` → 400), что и в остальных роутах. Ограничения по статусу
жизненного цикла нет — эндпоинт гейтится только `view_students`; у
не-студента просто не будет записей, и отчёт выйдет пустым.

### Правила агрегации

Один запрос на таблицу над множеством видимых групп (без N+1), свёртка в
JS — тот же приём, что `marked_count`/`graded_count` в роутах группы:

- **Посещаемость.** `attendance_percent = (present + late + excused) /
  marked × 100`, округление до целого, где `marked = present + absent +
  excused + late`. **`excused` и `late` засчитываются КАК присутствие —
  процент снижает только `absent`.** `null`, если `marked = 0`.
  Учитываются только неотменённые уроки (`is_cancelled = false`).
- **Средний балл.** Невзвешенное среднее `score / max_score × 100` по
  заданиям, где оценка выставлена; округление до одного знака. `null`,
  если оценок нет. Неоценённые задания в среднее не входят, но в
  `assessments[]` присутствуют со `score: null`.
- **Итог в `summary`** — средний балл считается как среднее **всех
  отдельных процентов** по всем видимым группам, а не как среднее из
  средних по группам (группа с большим числом оценок весит
  пропорционально). Итоговая посещаемость — `Σ(present+late+excused) /
  Σmarked × 100`.

### Доступ (пофильтр по группам, не общий 403)

Отчёт намеренно **мягче** гейта карточки студента. Два уровня:

1. **Верхний гейт** — как карточка: `view_students` на `department`
   студента (проходит `scope='all'` и подходящий `scope='department'`).
2. **Пофильтр по группам** — для КАЖДОЙ группы студента строка включается,
   только если есть `view_students` на target этой группы
   (`{ department_id, teacher_ids }`). Преподаватель (`scope='own'`) так
   видит строки только тех групп, где он значится в `class_teachers`.

403 возвращается только если **не пройден верхний гейт И не видна ни одна
группа**. Иначе — 200 с видимыми группами (список может быть пустым, если
верхний гейт пройден, но групп нет). В ответ попадают только видимые
группы и `visible_group_count` — число скрытых групп не раскрывается.

> **Расхождение для сведения.** Роуты `journeys` и серверная карточка
> студента `scope='own'` пока **не** реализуют (см. комментарии в
> `journeys/route.ts` и `journeys/[id]/route.ts`), поэтому сегодня
> преподаватель получает 403 на карточку студента целиком. Отчётный
> эндпоинт `scope='own'` учитывает (через пофильтр) и сам по себе готов к
> преподавателю — но вкладка «Успеваемость» живёт внутри карточки, а
> карточка (`students/[id]/page.tsx`) гейтит преподавателя раньше нового
> эндпоинта. Гейт карточки в этом заходе намеренно не менялся; чтобы
> преподаватель реально дошёл до вкладки, его нужно будет ослабить
> отдельно.

### Экран

Вкладка «Успеваемость» карточки студента. `LeadViewClient` (общий
компонент карточек лида/абитуриента/студента/выпускника) получил
необязательный проп `showReport`; вкладка добавляется **только** когда он
передан — так же, как вкладка «Учебный цикл» добавляется при
`studyLifecycle`. `students/[id]/page.tsx` передаёт `showReport` (эта
страница и так рендерится лишь для статусов учебного цикла). Прочие
карточки (лид/абитуриент/выпускник) проп не передают — их вид не меняется.
`StudentReportTab.tsx` грузит `GET .../report` и рисует: две плитки сводки
(общая посещаемость, средний балл), таблицу по группам (разбивка статусов
цветными бейджами, процент с покрытием `marked/total`, средний балл с
покрытием `graded/total`) и раскрывающуюся по клику на строку деталь
заданий (название, балл/макс, дата; неоценённое — прочерк). Состояния
загрузки, пустого списка и ошибки — на месте; при узкой колонке таблица
прокручивается горизонтально.

## Права и доступ

Используются существующие привилегии модуля `education`
(`lib/education/permissions.ts`, каталог засеян
`20260511175354_education_privileges.sql`):

- **`view_students`** — просмотр списка/детали уроков, посещаемости,
  заданий и оценок. Отдельной привилегии на просмотр оценок нет — экран
  «Оценки» использует то же право, что и «Журнал / уроки».
- **`set_lesson_topics`** — создание/правка/удаление уроков.
- **`mark_attendance`** — отметка посещаемости.
- **`set_grades`** — создание/правка/удаление заданий и выставление
  оценок.

Уроки/посещаемость и оценки не вводят новых привилегий — только
гарантируют, что нужным ролям выданы уже существующие коды:

- Системные роли `superadmin` / `tech_admin` / `campus_president`
  получают `mark_attendance` и `set_lesson_topics` со `scope='all'` —
  выдаёт блок 3 миграции `20260705150000` (идемпотентный паттерн `DO $$
  ... ON CONFLICT (role_id, module, privilege_code) DO UPDATE SET
  scope = 'all'`, тот же, что и в `20260705130000_alumni_graduation.sql`).
  Аналогично, `set_grades` со `scope='all'` для этих же трёх ролей
  выдаёт такой же идемпотентный блок миграции `20260705170000_grades.sql`.
  Все три роли уже имели `view_students` со `scope='all'` из более ранней
  миграции `20260511175354` (блок для системных ролей) — ни одна из двух
  новых миграций её не трогает.
- Роль **`teacher`** уже получила все четыре привилегии
  (`view_students`, `mark_attendance`, `set_lesson_topics`, `set_grades`)
  со `scope='own'` в блоке «4.8 teacher» миграции `20260511175354` — ни
  `20260705150000` (уроки/посещаемость), ни `20260705170000` (оценки)
  этот грант не создают и не изменяют, он был выдан заранее, одним
  блоком на все четыре кода.

Проверка `scope='own'` (см. `hasEducationPrivilege` в
`lib/education/permissions.ts`) означает: пользователь допущен к
действию, только если его `person_id` присутствует в `target.teacher_ids`
— то есть он значится в `class_teachers` конкретной учебной группы.
Преподаватель, не привязанный к группе, получит 403 на все эндпоинты
уроков, посещаемости и оценок этой группы, даже если у него есть эти
привилегии для других своих групп.

## Экраны

- `app/dashboard/education/class-groups/[id]/page.tsx` — серверная
  обёртка карточки группы. Строит `PrivilegeTarget` через
  `getClassGroupTarget`, затем четырьмя параллельными вызовами
  `hasEducationPrivilege` вычисляет флаги `canViewLessons`
  (`view_students`), `canManageLessons` (`set_lesson_topics`),
  `canMarkAttendance` (`mark_attendance`), `canSetGrades` (`set_grades`) и
  передаёт их клиенту вместе с производным `canViewGrades` (= значению
  `canViewLessons`, так как это одна и та же привилегия `view_students`).
  Если группа не найдена (или `id` не валиден) — все флаги остаются
  `false`, клиентский компонент покажет свой экран «группа не найдена».
- `ClassGroupCardClient.tsx` — клиентский компонент карточки (бывший
  `page.tsx` до вынесения серверной логики). Панель вкладок — «Обзор»,
  «Журнал / уроки» (только при `canViewLessons`), «Оценки» (только при
  `canViewGrades`); если ни одна из двух последних привилегий не выдана,
  вкладки вообще не показываются и виден только «Обзор» (тот же список —
  предметы, преподаватели, студенты, что и раньше).
- `LessonsJournalTab.tsx` — содержимое вкладки журнала:
  - таблица уроков (дата, время, тема, место, посещаемость `marked_count
    / enrolled_count`, действия), с бейджем «Отменён» для `is_cancelled`;
  - кнопка «Добавить урок» — только при `canManageLessons`; открывает
    модалку создания/редактирования (общий компонент для create/edit —
    режим определяется тем, передан ли существующий урок);
  - в каждой строке — кнопка «Посещаемость» (открывает `AttendancePanel`,
    видна всегда при `canViewLessons`) и, только при `canManageLessons`:
    «Изменить», «Отменить»/«Вернуть» (переключает `is_cancelled` через
    `PATCH`), «Удалить» (с `confirm()` и `DELETE`);
  - без `canManageLessons` строки полностью read-only — доступна только
    кнопка «Посещаемость».
- `AttendancePanel.tsx` — модалка посещаемости для одного урока: список
  записанных студентов, у каждого — четыре кнопки-тега статуса
  (present/absent/excused/late) с цветовой индикацией выбранного;
  кнопка «Отметить всех присутствующими» (только при `canMarkAttendance`);
  сохранение — `POST .../attendance` только с теми студентами, у кого
  выбран статус (значение `null` = «ещё не решено», не отправляется).
  Без `canMarkAttendance` все кнопки статуса неактивны, кнопки сохранения
  нет — только подсказка «Только просмотр — нет права отмечать
  посещаемость» и список текущих статусов.
- `GradesTab.tsx` — содержимое вкладки «Оценки»: матрица «студенты ×
  задания». Загружает список заданий (`GET .../assessments`), затем для
  каждого задания параллельно (`Promise.all`) — его оценки (`GET
  assessments/[id]/grades`); из объединённого ответа строится ростер
  (уникальные студенты по всем заданиям, отсортированные по имени) и
  карта оценок с ключом `` `${journey_id}|${assessment_id}` ``. Ячейка —
  оценка или «—», если оценки ещё нет. Первая колонка (студент) —
  `position: sticky`, чтобы оставаться видимой при горизонтальной
  прокрутке, когда заданий много (та же логика прокрутки, что в таблице
  уроков). Заголовок колонки задания — кликабельная кнопка (открывает
  `GradeEntryPanel`); под ней `/ max_score`, дата (если задана) и счётчик
  `graded_count / enrolled_count` (зелёный, если > 0). При `canSetGrades`
  под каждым заголовком — кнопки «Редактировать»/«Удалить» (последняя —
  с `confirm()`); кнопка «Добавить задание» над таблицей — тоже только
  при `canSetGrades`. Общая модалка создания/редактирования задания
  (`AssessmentFormModal`) — поля `title` (обязательно), `max_score`
  (по умолчанию 100), `assessment_date`, `description`.
- `GradeEntryPanel.tsx` — модалка выставления оценок для одного задания:
  список записанных студентов, у каждого — числовое поле (`min=0
  max={max_score} step=0.01`) и текстовое поле комментария. Клиентская
  валидация перед отправкой: пустое поле = «ещё не оценён» (в `entries`
  не попадает), иначе `0 ≤ score ≤ max_score` — при нарушении показывается
  баннер ошибки и `POST` не отправляется. Кнопка «Сохранить» (только при
  `canSetGrades`) отправляет `entries` только для тех студентов, у кого
  указан балл. Без `canSetGrades` все поля неактивны, кнопки сохранения
  нет — только подсказка «Только просмотр — нет прав на выставление
  оценок» и текущие значения.

Компонент использует только `useTranslations('education.report')` —
жёстко закодированных строк на экране отчёта нет.

## Побочные находки: FK hint для `persons`

При сквозной проверке (создание тестовой группы с преподавателем и
студентами) вскрылась ошибка PostgREST: `education_journeys` и
`class_teachers` имеют **по два** внешних ключа на `persons`
(`person_id`/`deleted_by` и `teacher_id`/`added_by` соответственно), из-за
чего неявный embed `person:persons(...)` в select-запросах падал с
`Could not embed because more than one relationship was found`. Ошибка
задевала не только новый роут `attendance`, но и три уже существовавших
роута группы — карточка группы (`class-groups/[id]/route.ts`), список
групп (`class-groups/route.ts`) и список записанных студентов
(`class-groups/[id]/enrollments/route.ts`); она была скрыта, пока таблица
`class_groups` была пуста на боевой БД. Исправлено явным указанием
FK-констрейнта в каждом select (`persons!class_teachers_teacher_id_fkey`
и `persons!applicant_profiles_person_id_fkey` — второе имя сохранилось от
переименования `applicant_profiles` → `education_journeys`, см.
`20260512162314_education_journeys_part1_create.sql`). Тот же именованный
хинт (`persons!applicant_profiles_person_id_fkey`) переиспользован в
роутах оценок (`assessments/[id]/grades`) при построении ростера — там
эта неоднозначность с самого начала обходится тем же способом.
