## 3. מה נבנה — הסברים ודוגמאות

### א. פורטל סטודנטים — אוכלוסיית משתמשים שנייה 🎓

עד היום המערכת הייתה לצוות בלבד. עכשיו **הסטודנטית עצמה** יכולה להתחבר
ולראות פורטל אישי — מבודד לחלוטין מצד הצוות. נבנה בשלושה שלבים (#105,
#107, #110):

- **התחברות והפרדה** — `/portal/login` (אימייל + סיסמה) → `/portal`.
  המפתח האבטחתי: **טבלת `student_credentials` נפרדת** מהתחברות הצוות.
  התחברות סטודנטית לעולם לא מגיעה לצד הצוות, והתחברות צוות לעולם לא
  מגיעה ל-`/portal`. שני עולמות, חומת אש ביניהם.
- **מה היא רואה** — לוח מחוונים אישי, לוח שנה, פגישות (קריאה בלבד),
  **הציונים והנוכחות שלה**, והודעות מהצוות אליה.
- **צד הצוות** — פאנל "התחברות סטודנט" בכרטיס (תחת `manage_students`)
  שיוצר/מאפס לה סיסמה ומציג אותה פעם אחת. וגם — **"תצוגה כסטודנטית"**
  (#94): הצוות רואה בדיוק מה שהיא רואה.

**למה זה משמעותי:** הוספת אוכלוסיית משתמשים היא מקום קלאסי לפרצות
אבטחה (סטודנטית שרואה נתונים של אחרת, או מגיעה לצד הצוות). הבחירה בטבלת
credentials נפרדת ובנתיב `/portal` מבודד היא הדרך הנכונה לעשות את זה.
SMS-login נדחה במודע (דורש ספק חיצוני) — החלטה טובה.

### ב. האצלת סמכויות §4 — "הזכות להעניק" 🔑

זה המהלך שהכי ריגש אותי, כי הוא מענה מדויק לשאלה שהעליתי אתמול. נבנתה
**מטא-הרשאה** `delegate_privileges`: מנהל יחידה יכול להעניק אותה לחבר
צוות (למשל סגן או מזכירה בכירה), וזה הופך אותו ל**מאציל** — הוא יכול
להעניק הרשאות לכפופים. אבל עם **שתי תקרות קשיחות**:

```ts
// app/api/education/units/[unitId]/members/[personId]/privileges/route.ts
// Cap делегирования (§4): не-глава (делегат) не может
// (а) пере-делегировать право delegate_privileges,
// (б) выдать больше, чем держит сам.
const isHead = isSuper || (await getHeadedUnitIds(session.person_id)).includes(params.unitId)
if (!isHead) {
  for (const code of toGrant) {
    // (א) נציג לא יכול להעניק מחדש את "זכות ההענקה" עצמה:
    if (code === 'delegate_privileges') return apiError('delegate_no_redelegate', 403)
    // (ב) נציג יכול להעניק רק הרשאה שיש לו בעצמו ביחידה:
    const holds = await hasEducationPrivilege(session, code, { department_id: params.unitId })
    if (!holds) return apiError('delegate_cap', 403)
  }
}
// ראש יחידה / superadmin — ללא הגבלה.
```

שתי השורות האלה הן בדיוק הכלל שהמלצתי עליו אתמול:
1. **אי אפשר להעניק מחדש את זכות ההענקה** — נציג לא יכול "לשכפל" מאצילים
   בלי ידיעת הראש. מונע שרשרת האצלה בלתי נשלטת.
2. **אי אפשר להעניק מה שאין לך** — נציג יכול להאציל רק סמכויות שהוא עצמו
   מחזיק. מונע הסלמת הרשאות.

הראש והsuperadmin פטורים מהתקרות. **זה מודל הרשאות בוגר ברמה שרואים
במערכות ארגוניות רציניות** — לא מובן מאליו בפרויקט בגיל הזה.

## 6. שורה תחתונה

יום שהוסיף למערכת **ממד חדש**: היא כבר לא רק כלי לצוות אלא פלטפורמה
דו-צדדית (צוות + סטודנטים), עם מודל האצלת סמכויות ברמה ארגונית אמיתית —
כולל בדיוק הכלל ("אי אפשר להאציל מה שאין לך") שהמלצתי עליו אתמול, מיושם
בקוד. האימות העצמאי (499/499 בדיקות, טיפוסים נקיים) מאשר יציבות.

**ההתקדמות: ~95%** (ומעל זה במונחי ה-scope המקורי; הפורטל הרחיב את
היעד). מה שנשאר הוא כמעט כולו **הכנה תפעולית** — הרצת מיגרציות, גיבוי,
ניטור, ובדיקת בידוד הפורטל. שבוע עד עשרה ימים של הכנה, לא פיתוח, מפרידים
מהשקה אמיתית ללקוח.
