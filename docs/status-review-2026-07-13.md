## 3. מה נבנה — הסברים ודוגמאות

### א. התראות — פעמון בזמן אמת 🔔

הפער שחזר בכל דוח מאז 06.07 ("יש עשרות תאריכי יעד ואף אחד לא מתריע")
— נסגר. נבנתה טבלת התראות **אוניברסלית** (לא ספציפית למקור אחד), כדי
שכל מודול יוכל לשלוח התראות בעתיד:

```sql
CREATE TABLE IF NOT EXISTS notifications (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  person_id  UUID NOT NULL REFERENCES persons(id) ON DELETE CASCADE,
  type       TEXT NOT NULL DEFAULT 'general',
  title      TEXT NOT NULL,
  body       TEXT,
  link       TEXT,          -- קליק → מנווט ישר לפריט
  metadata   JSONB NOT NULL DEFAULT '{}'::jsonb,
  read_at    TIMESTAMPTZ,   -- NULL = לא נקרא
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- אינדקס ייעודי לספירת "לא נקראו" מהירה:
CREATE INDEX idx_notifications_unread
  ON notifications(person_id) WHERE read_at IS NULL;
```

**התכנון הזהיר — deploy-safe.** ההערה במיגרציה מסבירה: הקוד קורא את
הטבלה **בהגנה** — לפני שהמיגרציה רצה, ה-API פשוט מחזיר רשימה ריקה,
ויצירת התראות היא best-effort. כלומר אפשר לפרוס את הקוד לפני שהטבלה
קיימת בלי לשבור כלום. זה בדיוק הפתרון לנקודת התורפה של "מיגרציה ידנית
+ פריסה אוטומטית" שהתרעתי עליה בדוחות קודמים.

המקור הראשון שמחובר: ועדת הקבלה — כשמועמדת מגיעה לשלב שדורש תפקיד מסוים,
כל בעל תפקיד רלוונטי מקבל התראה לצד המשימה האוטומטית.

### ב. העלאת קבצים אמיתית — Supabase Storage 📎

הפער השני: עד היום מודול "מסמכים" רק **רשם** מטא-דאטה — אי אפשר היה
להעלות את הקובץ עצמו. עכשיו יש העלאה אמיתית לתוך bucket **פרטי**, עם
צפייה דרך קישורים חתומים זמניים בלבד (לא גישה ציבורית):

```ts
// lib/documents/storage.ts
export const DOCUMENTS_BUCKET = 'documents'
export const MAX_UPLOAD_BYTES = 15 * 1024 * 1024   // 15MB

// רק PDF, תמונות ו-Word — נבדק בשרת, לא בדפדפן:
export const ALLOWED_MIME_TYPES = [
  'application/pdf', 'image/png', 'image/jpeg', 'image/webp',
  'application/msword',
  'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
] as const

// שם הקובץ מנוקה מתווים מסוכנים ומ-path traversal לפני שמירה:
function sanitizeFileName(name: string): string {
  const base = name.split(/[\\/]/).pop() ?? 'file'
  return base.replace(/[^\w.\-]+/g, '_').replace(/_+/g, '_').slice(0, 120) ...
}
```

**הנקודות החזקות:** הbucket פרטי (אין URL ציבורי לזליגה), ההעלאה
והצפייה עוברות רק דרך השרת עם בדיקת הרשאה (`documents.manage` להעלאה,
`documents.view` לצפייה), יש מגבלת גודל וסוג, ושם הקובץ מנוקה מפני
מתקפות path-traversal. זו הטמעה מאובטחת, לא "רק שיעבוד".

### ג. חתימות דיגיטליות על שלבי תהליך ✍️

יכולת חדשה משמעותית: כשעובד משלים שלב בתהליך (למשל אישור בוועדת קבלה),
הוא **חותם** עליו דיגיטלית — עם אפשרות להוסיף הערה/סיבה. נבנה בשכבות
(backend + אבטחה, ואז UI ללכידת החתימה). זה נותן שרשרת אחריות: מי אישר
מה, מתי, ולמה.

### ד. מודול חדש — יהדות (Jewishness verification) ✡️

מודול 19: סביבת עבודה לבירור יהדות של מועמדים, עם תפקידים ייעודיים
(`jewishness_officer`, `head_of_studies`). רלוונטי מאוד להקשר של מוסד
תורני. משתלב בתהליך הקבלה.

### ה. ועדת קבלה רב-שלבית 🏛️

תהליך הקבלה הורחב לוועדה **רב-שלבית**, שבה כל שלב חתום בידי בעל התפקיד
הנכון בלבד (עם אכיפת אבטחה — "שלב נחתם רק בידי התפקיד שלו"). נוספו גם
תורי עבודה (work-queues) ותזכורות לוועדה, ולוח סקירה לוועדה.

### ו. עוד שיפורי מוצר

- **יצירת סיסמאות אוטומטית** לחשבונות חדשים (במקום הזנה ידנית).
- **מסלולי לימוד** בכרטיס הסטודנט.
- **הודעה חוזרת לגיוס** על תוצאת הקבלה — סוגר את הלולאה בין התהליכים.
- **תיקון קריסה** בעמוד משימה ("Invalid time value") — באג אמיתי שנתפס ותוקן.

## 6. שורה תחתונה

היום שבו הרשימה "חובה לפני השקה" התרוקנה מפיתוח. שני הפערים הגדולים
(התראות, קבצים) נסגרו — ובצורה מאובטחת ומתוכננת (bucket פרטי, טבלה
deploy-safe), לא "רק שיעבוד". נוספו חתימות דיגיטליות, מודול יהדות, ועדת
קבלה רב-שלבית — כולם דרך תהליך PR מסודר. האימות העצמאי (430/430 בדיקות,
טיפוסים נקיים) מאשר יציבות.

**ההתקדמות: ~93%.** מה שנשאר הוא כמעט כולו הכנה להשקה — הרצת המיגרציות
של היום, גיבויים, ניטור, מדריכים וייבוא נתונים. כ-1–1.5 שבועות, ורובו
לא כתיבת קוד אלא הכנה תפעולית. המערכת עברה סופית מ"בפיתוח" ל"לקראת
השקה".
