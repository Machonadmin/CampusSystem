# Workflow Engine — анализ рисков отсутствия транзакций

> Диагностика без изменений кода, для совместного разбора. Покрывает все 5
> функций `lib/workflow/*.ts`. Каждая уже содержит комментарий в духе
> "Транзакций нет — частичное состояние при ошибке допустимо" — этот документ
> разбирает, что именно это означает на практике, и что можно с этим сделать.

## Общая картина

Ни одна из функций не откатывается при ошибке на середине. Но важнее другое:
**реальный failure mode — не громкий сбой, а тихое частичное состояние.**
Supabase JS клиент делает каждый `.insert()/.update()` отдельным HTTP-запросом
к PostgREST — своя мини-транзакция на каждый вызов. Если шаг 5 из 12 падает,
шаги 1–4 уже зафиксированы в БД, шаги 6–12 никогда не выполнятся — и это
выглядит не как ошибка, а как этап/процесс, который просто "завис" без задач,
без объяснения, без сигнала, что что-то пошло не так.

Это хуже громкого сбоя: пользователь получит `500`, попробует ещё раз — а
куратор через неделю просто не поймёт, почему у лида нет активных задач.

## Разбор по функциям (от простой к самой сложной)

### 1. `reactivateStage` — самая простая, но не безрисковая
1 update (`stage_instances` → active) → fire-and-forget событие → делегирует в
`createStartingTasks` (создаёт задачи циклом).

**Риск**: если `createStartingTasks` упадёт на второй задаче из трёх —
подэтап уже "active", часть задач создана, часть нет. Куратор видит подэтап
в работе, но не все ожидаемые задачи.

### 2. `startProcess` — создание процесса
create `process_instance` → цикл по этапам (create `stage_instance` для
каждого) → для активных этапов — событие (fire-and-forget) → если у этапа
есть задачи — `createStartingTasks`.

**Риск**: если цикл создания `stage_instances` падает на середине —
получаем `process_instance`, у которого физически не хватает части этапов.
Хуже: идемпотентность (проверка "уже есть активный instance") **не лечит**
это — повторный вызов найдёт уже сломанный instance и вернёт его как
`already_existed: true`, ничего не почини́в. Система не умеет само
восстанавливаться.

### 3. `handleTaskCompletion` — переходы между задачами
Цикл по `task_transitions`: для каждого перехода — проверка "не создана ли
уже" → (для `after_all`) проверка всех предшественников → создание задачи.

**Риск**: если создание задачи #2 из 3 падает — задача #1 создана, #3 нет.
Асимметричное продвижение веток подэтапа, которое трудно закрыть руками
(придётся разбираться, какая именно ветка "застряла").

### 4. `closeProcessEarly` — досрочное закрытие
~8 последовательных шагов: отменить активные подэтапы → события → отменить
задачи → закрыть `process_instance` → (если `convert_to_applicant`) обновить
journey.

**Риск**: если шаг "отменить задачи" падает после того, как подэтапы уже
помечены `cancelled`/процесс `completed` — получаем "зомби"-задачи: всё ещё
`pending`, но родительский подэтап и процесс уже мертвы. Задача выглядит
активной в UI, но принадлежит несуществующему процессу.

### 5. `completeStage` — самая сложная и самая рискованная
15+ последовательных операций, 3 крупные ветки логики:
- пометить подэтап `completed` → закрыть его задачи;
- **если финал закрывает процесс** (3b): отменить все прочие подэтапы →
  события → отменить все задачи → закрыть процесс → если
  `process_finish_reason='converted'` — обновить journey;
- **иначе** (обычный поток): найти исходящие переходы → для каждого — решить
  `after_one`/`after_all` → активировать целевой подэтап → событие → если у
  него есть задачи — создать их → затем отдельным проходом (5b) пометить
  недостижимые `waiting`-подэтапы как `skipped` → наконец проверить, не
  завершён ли весь процесс, и если да — закрыть его (и снова, отдельно,
  проверить `convert_to_applicant`).

**Риск**: самый широкий блаcт-радиус в движке. Конверсия лида в абитуриента
закодирована **в двух разных местах** этой функции (ветка 3b и финальная
проверка в шаге 6) с чуть разной логикой — если их в будущем поправят
по-разному, конверсия начнёт вести себя непоследовательно в зависимости от
того, какой веткой прошёл процесс. Плюс: если создание задач для
новоактивированного подэтапа #2 из 3 падает — тот же паттерн "подэтап active,
задач нет", помноженный на самую частую операцию в системе (продвижение
процесса — это то, что происходит каждый день).

## Почему это нельзя просто "обернуть в транзакцию" как раньше

Для `create_application`/`create_staff_member` перенос в один RPC был
относительно дёшев — это были линейные "создать A, потом B, потом C" без
ветвления. `completeStage` — это, по сути, обход графа с условной логикой
(`after_one`/`after_all`, каскадный skip недостижимых подэтапов). Перенести
это в PL/pgSQL один в один — не мелкая правка, а фактически переписывание
движка на другом языке, с высоким риском незаметно изменить поведение при
переносе. Это ровно тот класс работы, где нужен твой контроль на каждом
шаге, а не автономная работа.

**Дополнительная проблема**: в проекте нет тестов вообще (ни одного). Значит
"tsc прошёл, билд прошёл" — это не доказательство, что поведение не
изменилось. Переписывать самую сложную часть системы без тестов, которые
могли бы поймать регресс, — рискованно само по себе, независимо от вопроса
транзакций.

## Рекомендация — три параллельных, не обязательно последовательных, шага

**1. Дёшево и сразу: обнаружение аномалий, а не предотвращение.**
Простой SQL-запрос (или админский экран), который находит "подозрительные"
состояния уже сегодня: активные `stage_instances` с `has_tasks=true`, у
которых 0 задач; `process_instances` со статусом `completed`, у которых
остались незавершённые задачи. Это не чинит проблему, но делает её видимой
— вместо того, чтобы куратор случайно наткнулся на зависший лид через месяц.

**2. Начать перенос в RPC с самого простого, не с `completeStage`.**
`reactivateStage` (1 update + 1 цикл) — хороший первый кандидат: маленький
блаcт-радиус, легко проверить руками, что поведение не изменилось. Затем
`startProcess`. `completeStage` — в последнюю очередь, и только после того,
как появится хоть какой-то способ проверить, что поведение не сломалось
(руками через UI на тестовых лидах, если не тестами).

**3. Прежде чем трогать `completeStage` — минимальный ручной чек-лист
регрессии**, а не формальные тесты (это отдельная большая задача сама по
себе): 3–4 сценария (обычный переход, `after_all` с двумя ветками, финал с
`closes_process`, конверсия в абитуриента), которые можно прогнать руками на
тестовом лиде до и после изменения.

## Что не входит в эту рекомендацию

Не предлагаю оборачивать всё в PL/pgSQL "на всякий случай" — это
непропорционально дорого относительно текущего масштаба использования
движка. Не предлагаю останавливать текущую работу ради этого — это
отдельный трек, который стоит начать, когда будет время сесть вместе и
пройти хотя бы `reactivateStage` шаг за шагом.
