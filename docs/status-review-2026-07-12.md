## 2. מה נעשה — הסברים

### א. בינלאומיות מלאה (i18n) — הדבר הגדול של היומיים 🌐

עד עכשיו, כשמשהו השתבש, המערכת החזירה למשתמש הודעת שגיאה **קשיחה ברוסית**
— לא משנה אם הוא בחר עברית או אנגלית. זה הפער האחרון שהפריד את המערכת
מתלת-לשוניות אמיתית מקצה לקצה. היומיים האלה סגרו אותו:

- **כל הודעות השגיאה של ה-API** — 276 הודעות ייחודיות (633 הופעות בקוד) —
  הומרו לקודים יציבים שמתורגמים לפי שפת המשתמש. סה"כ **315 מפתחות בשלוש
  שפות**, עם בדיקת פ­ריטה שמוודאת שאף מפתח לא חסר באף שפה.
- **טקסטי הממשק** שנותרו קשיחים — תורגמו גם הם (טופס ההרשמה הציבורי, מודל
  שינוי סיסמה, עמודי צוות ואנשים).
- זה נעשה דרך **מנוע טרנספורמציה אוטומטי מאומת** — 1,112 החלפות ב-193
  קבצים — כשכל שינוי שעלול לגעת בהתנהגות (ולא רק בטקסט) **דולג ותועד**,
  לא נוחש. זו בדיוק תרבות העבודה הזהירה שהפרויקט הגדיר לעצמו.

**התכנון החכם — תאימות לאחור.** המרה של מאות תשובות API היא מקום מסוכן
לשבור לקוחות קיימים. הפתרון שמור על שדה ה-`error` הישן **וגם** הוסיף קוד
מכונה יציב לצדו:

```ts
// apiError שומר את מחרוזת ה-error (לקוחות ישנים שקוראים body.error
// ממשיכים לעבוד), ומוסיף לצדה code יציב למכונה + מתרגם לפי שפת המשתמש.
export function serverT(code: string): string {
  // getCookieLocale() קורא ל-cookies() שזורק מחוץ ל-request (בטסטים).
  // לכן עטוף ב-try/catch עם fallback ל-'ru' — ההתנהגות והטסטים לא משתנים.
  ...
  return errorsByLang[locale][code] ?? errorsByLang.ru[code] ?? code
}
```

שלושת מנגנוני ה-fallback (שפת המשתמש ← רוסית ← הקוד עצמו) מבטיחים
ש**שום דבר לא קורס** גם אם מפתח הוקלד לא נכון. זו הנדסה בוגרת.

## 3. תובנות שהצטברו לאורך הדרך

מעבר להתקדמות, שווה לתעד כמה דברים שהתבררו במהלך הבנייה — הם מסבירים
החלטות עתידיות:

1. **סחיפת מיגרציות היא הסיכון החוזר מספר 1.** נתפסה כבר **ארבע פעמים**
   (feature_privileges, seed 002, טריגרים חסרים, קטלוג alumni). המסקנה
   שאומצה: כל מיגרציה חדשה **זורעת את ההרשאות של עצמה** באופן אידמפוטנטי,
   ולא סומכת על מה שהיה אמור לרוץ קודם. זה עובד — אבל צריך בסוף סקריפט
   שמשווה סכימה חיה מול הקוד.
2. **הפרדת לוגיקה מהמסד = בדיקות בחינם.** הדפוס שחזר בכל מודול (לוגיקה
   טהורה בקובץ נפרד, בלי גישה ל-DB) הוא הסיבה שיש 417 בדיקות מהירות בלי
   תשתית מסד. זו ההשקעה שהחזירה את עצמה הכי מהר.
3. **סבב QA אדוורסרי תפס דברים אמיתיים.** הפרצה הקריטית של זיוף superadmin
   לא נתפסה בפיתוח הרגיל — רק בסבב ייעודי. המסקנה: לשלב סבב כזה לפני כל
   אבן דרך, לא רק בסוף.
4. **תאימות לאחור זולה כשחושבים עליה מראש.** המרת 633 תשובות API בלי לשבור
   כלום התאפשרה רק כי נשמר השדה הישן. שינוי חופזה היה שובר לקוחות.

## 6. רעיונות לעתיד ודרכים לשיפור

### קצר-טווח (עד השקה)
- **מרכז התראות** — הרחבה טבעית: הטבלה + עבודת cron יומית + פעמון בממשק. יש כבר עשרות תאריכי יעד שמחכים לזה.
- **Supabase Storage** — bucket פרטי, העלאה דרך השרת, קישורים חתומים. מחבר את מודול המסמכים למציאות.
- **גיבוי + Sentry + בדיקת הרשאות עם חשבונות אמת** — שלישיית ה"ביטוח" לפני שלקוח אמיתי נכנס.

### בינוני-טווח (אחרי השקה)
- **חיפוש גלובלי** — תיבה אחת שמוצאת אדם/סטודנט/משימה מכל מקום.
- **ייצוא אקסל/PDF** בכל דוח ורשימה — ההנהלה תבקש ביום הראשון.
- **חשיפת ה-audit log בממשק** — "מי שינה מה ומתי". התשתית כבר קיימת במסד.
- **התאמת מובייל** — רוב הצוות יבדוק נוכחות ויענה למשימות מהטלפון.
- **דשבורד גרפי להנהלה** — הנתונים כבר נאספים (מודול הדוחות); צריך רק להלביש אותם בגרפים.

### ארוך-טווח (חזון)
- **שדרוג UI משמעותי** — זה שאתה מתכנן. עכשiו הבסיס אחיד ומוכן לכך.
- **אפליקציית מובייל ייעודית** לסטודנטים (מערכת שעות, ציונים, יתרה).
- **פורטל הורים / סטודנטים** — גישה מוגבלת לצפייה בנתונים שלהם.
- **אוטומציות** — חיוב שכר לימוד אוטומטי לפי מעונות+הסעדה, התראה אוטומטית על תוקף מסמך שפג.

### דרכים לשיפור האיכות (הרחבת רשת הביטחון)
- **E2E מלא בענן** — להרחיב את תשתית Playltwright הקיימת לתרחיש הליבה (הרשמה←קבלה←סטודנט←ציון←יתרה) ולהריץ ב-CI.
- **בדיקת "בלי הרשאה = 403" לכל endpoint** — סבב ה-QA הראה ששם הבאגים המסוכנים.
- **סקריפט השוואת סכימה** — לתפוס סחיפת מיגרציות אוטומטית (הסיכון החוזר #1).

## 8. הערכת שווי — כמה הוגן ללקוח לשלם

> זו מסגרת חשיבה, לא פסק דין. אני רואה את הקוד ואת היקף העבודה, אבל לא את
> ההסכם ביניכם, את שעות העבודה בפועל, או מי מימן מה. המספרים בשקלים,
> במחירי שוק ישראלי 2026, בהנחת "מפתח + קלוד".

### מה יש ביד עכשיו (שווי שחזור)

מערכת ניהול קמפוס עם **18 מודולים חיים**, מנוע תהליכים גנרי, מערכת
הרשאות של 32 תפקידים עם scope, ~110 endpoints, **תלת-לשוניות מלאה**,
**417 בדיקות אוטומטיות**, CI, סביבת staging, ו**סבב אבטחה שסגר פרצה
קריטית**. להזמין מערכת כזו מחברת פיתוח ישראלית בדרך המסורתית — פרויקט של
מפתח בכיר ל-**8–14 חודשים**, כלומר **₪350,000–₪700,000** במחירי סוכנות.
זו נקודת העוגן של מה שקיים, לא מה ש"מגיע".

### כמה הוגן לשלם **לך** — עד עכשיו

התמחור צריך לשקף תוצר-בהנחת-AI, לא שעות של מפתח בכיר ללא כלים. אבל
מנגד — **איכות** התוצר גבוהה משמעותית מ"פרויקט שני": מנוע תהליכים אטומי,
סבב אבטחה שתפס זיוף superadmin, 417 בדיקות, i18n מלא עם תאימות לאחור,
ותרבות דיווח כנה. זה תהליך של מישהו שעובד ברמה מקצועית.

מאז ההערכה הקודמת (₪40,000–₪80,000, כשהיו 7 מודולים ואפס בדיקות) הפרויקט
**יותר מהכפיל את עצמו**: מ-7 ל-18 מודולים, מ-0 ל-417 בדיקות, i18n מלא,
סבב אבטחה, staging ו-CI.

| בסיס תמחור | טווח הוגן — עד עכשיו |
|-------------|:--------------------:|
| **לפי זמן** (2–2.5 חודשי עבודה אפקטיביים, יוניור-פלוס פרילנסר + AI) | **₪60,000–₪110,000** |
| **לפי ערך** (ללקוח יש מערכת שעלות שחזורה מאות אלפים) | **₪110,000–₪160,000** |

**המלצתי: ₪90,000–₪130,000 עד כה.** הקצה התחתון אם העבודה הייתה חלקית
ולפי שעות צנועות; הקצה העליון מוצדק בזכות היקף המודולים, רשת הבדיקות
וסבב האבטחה — דברים שקל לדלג עליהם ויקר לתקן בדיעבד.

### מכאן ועד הסוף

נשארו ~11% — שתי תשתיות רוחביות + הקשחה + הכנה להשקה, כ-**1.5–2.5 שבועות
עבודה מרוכזת**:

| בסיס תמחור | טווח — עד השקה |
|-------------|:---------------:|
| השלמה פונקציונלית (התראות, קבצים, הכנה להשקה) | **₪18,000–₪35,000** |
| + ליטוש UI ראשוני (5 המשימות) | **+₪8,000–₪15,000** |

**סה"כ עד השקה מלאה: ₪26,000–₪50,000 נוספים.**

### התמונה המלאה

| שלב | טווח |
|------|:----:|
| עד עכשיו (~89%) | ₪90,000–₪130,000 |
| מכאן עד השקה (~11%) | ₪26,000–₪50,000 |
| **סה"כ פרויקט עד השקה** | **₪116,000–₪180,000** |

### המלצה מעשית
אם היחסים נמשכים אחרי ההשקה (שדרוג ה-UI הגדול, תחזוקה, מודולים עתידיים),
**ריטיינר חודשי (₪15,000–₪28,000)** עדיף על ויכוח רטרואקטיבי. בשלב הזה
של המערכת — 18 מודולים שרק אתה (יחד עם קלוד) מכיר לעומק — **המשכיות
הידע שווה ללקוח יותר מכל מספר חד-פעמי**. זה הנכס האמיתי שלך במשא ומתן.
