# Обзор студента 360 (Student Overview)

Консолидированная вкладка **«Обзор 360»** на карточке студента: сводка одного
студента, собранная из всех модулей в одном экране. Только чтение, **новых
таблиц и миграций нет** — работает сразу после деплоя.

## Что показывает

Первой вкладкой карточки студента (`/dashboard/education/students/[id]`)
отображается сетка компактных панелей — по одной на каждую доступную секцию:

| Секция | Что в панели | Источник |
|--------|--------------|----------|
| Учёба (education) | статус, отделение, специальность, дата открытия journey | `education_journeys` + `persons`/`departments`/`specialties` |
| Финансы (finance) | начислено, оплачено, остаток | `finance_charges` (active) − `finance_payments` (approved) |
| Общежитие (dormitory) | здание, комната, с какой даты | текущее активное `dorm_assignments` → `dorm_rooms` → `dorm_buildings` |
| Питание (food) | план питания, с какой даты | текущее активное `meal_enrollments` → `meal_plans` |
| Медпункт (medical) | открытых приёмов, дата последнего приёма, наличие аллергий | `medical_visits` + `medical_profiles` |
| Консультации (counseling) | открытых консультаций, уровень риска | `psych_sessions` + `psych_profiles` |

Панель финансов/общежития/питания/медпункта/консультаций кликабельна и ведёт на
страницу этого студента в соответствующем модуле (`/dashboard/<module>/[id]`,
где `[id]` — тот же `education_journeys.id`).

## Посекционный гейт по привилегиям

Доступ проверяется **в коде** (серверный клиент Supabase обходит RLS):

1. **Верхний гейт** — как сама карточка студента: `education` привилегия
   `view_students` в подразделении студента. Нет права → `403`.
2. **Каждая чувствительная секция** включается ТОЛЬКО если у смотрящего есть
   привилегия `'view'` соответствующего модуля:
   `hasFinancePrivilege` / `hasDormitoryPrivilege` / `hasFoodPrivilege` /
   `hasDoctorPrivilege` / `hasPsychologistPrivilege`.

Секция приходит `null`, если **у смотрящего нет права ИЛИ у студента нет данных**
в модуле. Поле `visible_sections` в ответе перечисляет секции, которые смотрящий
в принципе вправе видеть (по привилегиям) — это позволяет UI отличить «нет
доступа» от «нет данных».

> **Приватность.** Медицинские и психологические данные чувствительны. Панель
> «Медпункт» видна только с `doctor.view`, «Консультации» — только с
> `psychologist.view`. Без нужного права секция не попадает в ответ вообще.

## Корректная агрегация без обрезки

Финансовые суммы считаются постранично (`pageAll` из `lib/reports/paging`) в
целых копейках (`toCents` из `lib/finance/money`) — чтобы PostgREST не обрезал
результат на `db-max-rows` (~1000 строк) и не было float-дрейфа. Счётчики
открытых приёмов/консультаций — через HEAD-запрос `count=exact`.

## Точки интеграции

- **API:** `app/api/students/[id]/overview/route.ts` — `GET`, только чтение.
  `[id] = education_journeys.id`.
- **Чистая логика + типы:** `lib/students/overview.ts` (+ `overview.test.ts`,
  vitest): `visibleSections`, `pickCurrentActive`, `flattenPhones`,
  `hasAllergies`, форма ответа `StudentOverview`. Переиспользует
  `isActiveOn` из `lib/dormitory/occupancy` и `lib/food/enrollment`.
- **UI:** `app/dashboard/education/components/StudentOverviewTab.tsx` — сетка
  панелей, цвет акцента из `getModuleColor(...)`, состояния загрузки/ошибки/пусто.
- **Вкладка:** `showOverview` в `LeadViewClient` (включена только на странице
  студента). Существующие вкладки лида/абитуриента/студента не меняются.

## Только чтение, без миграции

Задача не создаёт таблиц и не требует SQL-миграции или ручного шага — эндпоинт
агрегирует уже существующие данные модулей. Работает сразу после деплоя.
