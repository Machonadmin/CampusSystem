## חלק ג' — סקירת המבנה ההיררכי (הבקשה המרכזית)

תיארת את היעד: **מוסד שמנהל אחד בראשו → ענפים, לכל ענף אחראי → מזכירות
→ מורים → תלמידים**, עם האצלת סמכויות בכל שכבה. בדקתי כמה הקוד באמת בנוי
ככה. הבשורה הטובה: **רוב המבנה כבר קיים.** הפחות טובה: יש פער אדריכלי
אחד מהותי שאתה צריך להבין ולהחליט לגביו.

### מה כבר קיים — והוא טוב

**1. העץ הארגוני קיים בנתונים.** טבלת המחלקות בנויה כעץ מהיום הראשון:

```sql
CREATE TABLE departments (
  id UUID PRIMARY KEY,
  name TEXT NOT NULL,
  parent_id UUID REFERENCES departments(id),   -- ← עץ היררכי
  head_person_id UUID REFERENCES persons(id),   -- ← ראש המחלקה/ענף
  ...
);
```

`parent_id` נותן עומק בלתי מוגבל (מוסד → ענף → תת-ענף → כיתה),
ו-`head_person_id` מגדיר מי הראש של כל צומת. זה בדיוק השלד שתיארת.

**2. אדם אחד, כמה כובעים.** `staff_positions` מאפשר לאדם להיות בכמה
תפקידים במחלקות שונות, עם דגל `is_head`:

```sql
CREATE TABLE staff_positions (
  person_id UUID NOT NULL REFERENCES persons(id),
  department_id UUID NOT NULL REFERENCES departments(id),
  position_ru TEXT NOT NULL,
  is_head BOOLEAN DEFAULT FALSE,   -- ← ראש היחידה הזאת
  ...
);
```

**3. מודל scope תלת-שכבתי — לב האצלת הסמכויות.** לכל הרשאה יש טווח:

```ts
export type Scope = 'all' | 'department' | 'own'
// all = כל המוסד · department = היחידה שלי · own = רק שלי
```

וכשלאדם כמה תפקידים עם טווחים שונים — **הטווח הגבוה מנצח** (`reduceScopes`).
זה נכון בדיוק למנהל שהוא גם ראש ענף (department) וגם חבר ועדה (all).

**4. האצלת סמכויות אישית — הפיצ'ר החדש והחשוב (#49).** המנגנון
שהמנהל "מאציל" בו סמכות לאדם ספציפי מעבר לתפקידו:

```ts
// lib/permissions/scope.ts — person_privileges מעל תפקידים
export function applyPersonGrants(base, grants) {
  for (const g of grants) {
    if (!g.is_granted) { delete out[code]; continue }  // ← שלילה מפורשת
    if (out[code] !== 'all') out[code] = 'department'    // ← הענקה אישית
  }
}
```

זה בדיוק "המנהל מאציל": מזכירה מקבלת בבסיס רק צפייה ביחידתה, והמנהל
מעניק לה אישית עוד סמכויות — בלי לשנות את התפקיד הגלובלי. יש גם **שלילה
מפורשת** (`is_granted=false`) שגוברת על התפקיד. זה מודל בוגר.

**5. המבנה שסוכם איתך כבר ממומש** (מהמיגרציה `20260715120000`):
> קודש — מנהל אחד לכולם. חול — מנהל לכל יחידה (תיכון/קולג'/אוניברסיטה/
> טורו). מעליהם "ראש חול". כל מנהל: המזכירות שלו + המורים שלו.

### ⚠️ הפער המרכזי שסוקר חייב להצביע עליו: **הטווח שטוח, לא היררכי**

זו הנקודה הכי חשובה בדוח. מנוע ההרשאות בודק שייכות למחלקה כך:

```ts
// lib/permissions/scope.ts — grantsAccess()
if (scope === 'department') {
  if (!target?.department_id) return true
  return ctx.departmentIds.includes(target.department_id)   // ← התאמה מדויקת
}
```

`ctx.departmentIds` = המחלקות שהאדם משויך אליהן **ישירות**. הבדיקה היא
`includes` — התאמה מדויקת. **המנוע לא מטייל בעץ ה-`parent_id`.**

**מה זה אומר בפועל:** אם ל"ראש חול" יש תפקיד בענף "חול", והוא רוצה לראות
תלמיד בתת-היחידה "קולג'" (שהיא צאצא של "חול" בעץ) — **הגישה תיכשל**,
אלא אם יש לו תפקיד מפורש גם ב"קולג'". ההיררכיה קיימת בנתונים אבל **לא
מיושמת בהחלטת ההרשאה**.

**איך זה נפתר כרגע:** במקום שהעץ יעשה את העבודה, נתנו ל"ראש חול" תפקיד
ראש **בכל יחידה בנפרד** (ציטוט מהמיגרציה: "моделируется как studies_manager
с позициями-главой во ВСЕХ единицах"). זה עובד — אבל זה **workaround
ידני**, לא ירושה אוטומטית.

**למה זה חשוב לך:** ככל שהמוסד יגדל (עוד ענפים, עוד עומק), כל מנהל בכיר
יצטרך שיוך ידני לכל תת-יחידה. אם תוסיף מחר תת-ענף חדש תחת "חול", תצטרך
לזכור לשייך אליו את ראש חול ידנית — אחרת הוא "לא רואה" אותו. זה מקור
לטעויות שקטות.

**זה לא באג — זו בחירה אדריכלית.** והיא לגיטימית לגודל הנוכחי. אבל אתה,
כבעל המוצר, צריך להחליט במודע: להישאר עם שיוך מפורש (פשוט, אבל ידני),
או להשקיע בטווח היררכי אמיתי (המנוע יטייל בעץ — מנהל בצומת רואה אוטומטית
את כל מה שתחתיו). ההשקעה בינונית (~2–3 ימים), והנקודה היחידה שצריך לגעת
בה היא `grantsAccess`.

## חלק ד' — מה אתה צריך להבין ולהחליט בעצמך (שאלות בעלים)

סוקר טוב לא רק מוצא פערים — הוא מזקק את ההחלטות שרק אתה יכול לקבל. אלה
השאלות שכדאי שיהיו לך תשובות ברורות עליהן, כי הקוד ימשיך להיבנות סביבן:

1. **ירושה בעץ — כן או לא?** (הפער מחלק ג'.) האם מנהל בכיר צריך לראות
   אוטומטית את כל תת-היחידות שתחתיו, או שאתה מעדיף שיוך מפורש? זו החלטה
   שמשפיעה על כל האצלת סמכויות עתידית.

2. **"ראש חול" — אחד או רבים?** המיגרציה מציינת "пока один; может измениться"
   (בינתיים אחד, עשוי להשתנות). אם בעתיד יהיו כמה — המודל הנוכחי (תפקיד
   בכל יחידה) יסתבך. כדאי להחליט עכשיו.

3. **מי מאציל למי — ומה הגבול?** כרגע האצלה אישית (`person_privileges`)
   היא כלי חזק. האם **כל** מנהל יכול להאציל, או רק דרגות מסוימות? האם
   מנהל יכול להאציל סמכות שאין לו בעצמו? כדאי שיהיה כלל ברור.

4. **מורה מול מזכירה — מי רואה ציונים?** המיגרציה הסירה `set_grades`
   מהמורה ("оценки пока только руководитель" — ציונים בינתיים רק המנהל).
   זו החלטה מדיניות שכדאי שתהיה מודעת לה — היום המורה **לא** יכול לתת
   ציון אלא אם המנהל האציל לו אישית.

5. **קודש מול חול — עד כמה נפרדים?** יש שיבוץ דו-תחומי. האם תלמיד יכול
   להיות פעיל בשניהם בו-זמנית? מי רואה את התמונה המלאה שלו — רק מנהל
   שמעל שניהם, או כל אחד את החלק שלו?

> ההמלצה שלי: כתוב לעצמך מסמך של עמוד אחד — "המבנה הארגוני של המוסד"
> — עם התשובות. זה יהפוך את כל ההחלטות העתידיות למהירות ועקביות, ויהיה
> מקור האמת שגם אני וגם כל מפתח עתידי נבנה לפיו.

## חלק ו' — שורה תחתונה

יומיים חזקים שבנו בדיוק את מה ששאלת עליו: המבנה ההיררכי. **הבשורה: השלד
כבר קיים ובנוי נכון** — עץ מחלקות עם ראשים, אדם עם כמה תפקידים, טווח
תלת-שכבתי, והאצלה אישית מהמנהל (person_privileges). זה מודל בוגר של
"מנהל שמאציל סמכויות".

**הממצא היחיד שדורש את תשומת לבך:** המנוע שטוח, לא היררכי — הוא בודק
שייכות ישירה למחלקה ולא מטייל בעץ. זה עובד היום דרך שיוך מפורש, אבל
אתה צריך להחליט במודע אם זו האסטרטגיה לטווח ארוך (חלק ד', שאלה 1).

**ציון כולל: 83/100** — מוצר חזק, לקראת השקה. מה שמפריד מ-90+ הוא לא
פונקציונליות (יש בשפע) אלא **מוכנות תפעולית** — גיבוי, ניטור, ייבוא —
שזו העבודה הבאה. האימות העצמאי (494/494 בדיקות, טיפוסים נקיים) מאשר
שהבסיס יציב לחלוטין.

**ההתקדמות: ~93% → ~94%.** מכאן זו כבר לא בנייה אלא הבשלה והכנה להשקה.
