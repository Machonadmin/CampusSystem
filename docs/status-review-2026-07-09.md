## 2. מה נוסף — הסברים

### א. כל המודולים הושלמו — 18/18 ✅

שני המודולים האחרונים שנותרו נבנו:

- **תורמים (Sponsors)** — ספריית תורמים + פנקס תרומות (מי התחייב, מה
  התקבל). זה צד ההכנסות שהשלים את מודול הכספים.
- **אבטחה (Security)** — יומן אירועי קמפוס עם workflow סטטוסים (דיווח ←
  טיפול ← סגירה).

בנוסף נבנו שני דברים שלא היו ברשימת המודולים המקורית אבל מוסיפים ערך אמיתי:

- **ספריית אנשים (People / persons)** — מדריך לצפייה בצוות ובסטודנטים
  במקום אחד, עם הרשאות. הפך למודול מן המניין (נוסף ל-`IMPLEMENTED_MODULES`).
- **לוח שנה אישי (Calendar)** — לוח בסגנון Google לכל עובד: הפגישות שלו,
  שיעורים שהוא מלמד, משימות, וסלוטים חוזרים — במבט אחד. תוכנן חכם:
  **self-scoped** (כל אחד רואה רק את עצמו) בלי מנגנון הרשאות מורכב, אלא
  בפילטר פשוט `provider_id = session.person_id` בכל שאילתה. אפילו יש
  תמיכה בתאריכים עבריים ובימי הולדת כאירוע שנתי.

### ב. סבב QA ואבטחה — הדבר החשוב של היום 🛡️

זה מה שמייחד את היומיים האלה. במקום להוסיף עוד ועוד, המערכת עברה **ביקורת
אבטחה שיטתית** שמצאה ~15 ליקויים אמיתיים ותיקנה אותם. אלה לא ליקויים
קוסמטיים — כמה מהם היו חמורים. הבולטים:

**#1 (קריטי) — מפתח JWT שניתן לנחש.** בקוד הישן היה fallback: אם משתנה
הסביבה `JWT_SECRET` חסר, המערכת השתמשה בערך ברירת מחדל ידוע. משמעות: מי
שיודע את הערך הזה יכול **לזייף session של superadmin** ולהשתלט על המערכת.
התיקון — "fail-closed": בפרודקשן המערכת פשוט מסרבת לעלות אם אין מפתח חזק
ואמיתי.

```ts
// lib/auth/config.ts — QA FIX #1
export function getJwtSecret(): string {
  const secret = process.env.JWT_SECRET
  if (!secret || secret.length < 32 || secret === DEFAULT_INSECURE_SECRET) {
    if (process.env.NODE_ENV === 'production') {
      throw new Error('JWT_SECRET is missing, too short, or insecure default...')
    }
    return DEFAULT_INSECURE_SECRET  // dev/test בלבד
  }
  return secret
}
```

**#4 — חשיפת נתונים אישיים רגישים (PII).** התגלה שכל מי שיש לו הרשאת
צפייה באנשים ראה גם דרכון, כתובת, מצב משפחתי ותאריך לידה. אבל בספריית
אנשים משתמשים הרבה תפקידים (רופא, מורה, רכז...) שלא אמורים לראות דרכון.
התיקון — הרשאה נפרדת `persons.view_sensitive`, שניתנת רק לקבוצה מצומצמת
(מנכ"ל, משאבי אנוש, אדמין); לכל השאר השדות הרגישים פשוט מוסתרים.

**#3 — "fail-closed" בהרשאות מבוססות-מחלקה.** כשלמשתמש יש הרשאה מוגבלת
למחלקה שלו, והיעד (למשל סטודנט) בלי מחלקה משויכת — הקוד הישן **התיר**
את הפעולה. שונה ל**מנוע** (ברירת מחדל בטוחה).

**#12 — מניעת race condition בכספים (anti-TOCTOU).** אישור תשלום ועדכון
חיוב הפכו לכתיבות אטומיות מותנות — אם שני אנשים מנסים לאשר את אותו תשלום
בו-זמנית, השני מקבל שגיאה 409 במקום כפילות.

ועוד: עימוד (pagination) בכל הרשימות והאגרגציות הגדולות (כדי שרשימה של
1,000+ לא תישבר), הסרת לוגים שחשפו אימייל/תקינות סיסמה, ותיקון הרשאות
נקודתיות במשימות ומסמכים.

**למה זה חשוב לך כלקוח:** מערכת שמנהלת נתוני סטודנטים, כספים ותיקים
רפואיים חייבת לעמוד בסטנדרט אבטחה. היומיים האלה העלו את המערכת מ"עובדת"
ל"עומדת בביקורת אבטחה". זה בדיוק סוג העבודה שקל לדלג עליה ויקר לתקן
אחר כך.

### ג. תשתית גישה לפי תפקיד

נוספה מיגרציית יסוד `role → module access`: הבסיס לכך שכל תפקיד רואה בדיוק
את המודולים הרלוונטיים לו. כבר עכשיו — עמוד הבית מציג לכל משתמש רק את
המודולים שיש לו גישה אליהם (במקום הכול לכולם).

## 5. איך להמשיך מכאן בלי להתפזר — ה-UI ברמת התצוגה ללקוח

הבנתי אותך: **לא** רפורמת UI גדולה עכשיו (זה יבוא בהמשך), אלא ליטוש
תצוגה בסיסי כדי שהמערכת תיראה מוצגת ומהוקצעת ללקוח. הנה מסגרת ממוקדת —
**לא 30 רעיונות, אלא כלל אחד ו-5 משימות בלבד.**

### הכלל שמונע התפזרות
> **"אחידות לפני יופי."** לא מוסיפים שום אלמנט חדש. עוברים על מה שכבר
> קיים ומוודאים שכל 18 המודולים נראים כמו **אותה מערכת** — אותם רכיבים,
> אותם מרווחים, אותן מצבי-ריק. משקיעים שבוע, לא יותר, ולא נוגעים
> בארכיטקטורה.

### 5 המשימות (בסדר הזה, יום כל אחת לכל היותר)

1. **רכיבי-על אחידים (Design tokens בסיסי).** להגדיר פעם אחת: כרטיס,
   כפתור ראשי/משני, תג-סטטוס (צבע לפי מצב), כותרת-עמוד. להחליף בכל
   המודולים. תוצאה: הכל נראה מאותה משפחה. *(כבר יש `module-colors.ts` —
   בסיס מצוין להרחיב ממנו.)*
2. **מצבי ריק, טעינה ושגיאה.** לכל טבלה/רשימה: "אין עדיין נתונים" ידידותי
   (במקום טבלה ריקה), spinner אחיד בטעינה, והודעת שגיאה קריאה. זה הדבר
   שהכי משדר "מערכת גמורה".
3. **טבלאות עקביות.** אותו עיצוב שורות, אותו מיקום חיפוש/פילטרים, אותה
   התנהגות עמודים (pagination — כבר קיים בצד השרת מסבב ה-QA), שמות
   סטודנטים תמיד קליקביליים. פעולה אחת שחוזרת ב-18 מקומות.
4. **טפסים ומשוב.** אחרי כל פעולה — הודעת הצלחה/כישלון קצרה (toast).
   ולידציה שמראה למשתמש **מה** לא תקין ליד השדה. כפתורים שמראים מצב
   "שולח...". מונע את התחושה ש"לחצתי וכלום לא קרה".
5. **ניווט וכותרות.** breadcrumb פשוט (איפה אני), כותרת ברורה לכל עמוד,
   ו-active-state בסרגל הצד. שהמשתמש תמיד ידע איפה הוא.

### מה **לא** לעשות עכשיו (הרשימה שמונעת התפזרות)
אנימציות, ערכות נושא, dark mode, דשבורדים גרפיים מושקעים, גרפים
אינטראקטיביים, redesign של הכרטיסים, ספריית קומפוננטות חיצונית. **כל
אלה שייכים ל"שינוי ה-UI המשמעותי" העתידי — לא עכשיו.** אם רעיון UI לא
נמצא ב-5 המשימות למעלה, הוא נכנס לרשימת "גרסה 2" ולא נוגעים בו.

## 6. רעיונות כלליים — לשיפור, הוספה ובדיקה

מחולק לפי אופק זמן, כדי שיהיה קל להחליט מה עכשיו ומה אחר כך.

### לפני השקה (חשוב באמת)
- **גיבוי ושחזור.** לוודא ש-Supabase מגבה יומית, ולתרגל שחזור אחד מלא ל-staging. זה הביטוח היחיד מפני אסון נתונים.
- **ניטור שגיאות (Sentry).** כשמשהו נשבר אצל משתמש — לראות את זה מיד, בלי לחכות לתלונה.
- **בדיקת הרשאות מקצה-לקצה עם חשבונות אמת.** ליצור חשבון בכל תפקיד מרכזי ולוודא ידנית שכל אחד רואה בדיוק מה שמותר לו (סבב ה-QA תיקן את הלוגיקה; כדאי לאשש עם משתמשים אמיתיים).
- **מדיניות סיסמאות ונעילת חשבון** אחרי כמה ניסיונות כושלים.

### אחרי השקה (הרחבות ערך)
- **חיפוש גלובלי** — תיבת חיפוש אחת שמוצאת אדם/סטודנט/משימה מכל מקום.
- **ייצוא לאקסל/PDF** בכל הדוחות והרשימות (הנהלה תבקש את זה מיד).
- **מרכז התראות + אימייל** — הרחבה טבעית של מודול ההתראות כשייבנה.
- **היסטוריית פעולות למשתמש** ("מי שינה מה ומתי") — כבר יש audit_log בבסיס; לחשוף אותו בממשק.
- **תמיכה במובייל** — לוודא שהתצוגה סבירה בטלפון (רוב הצוות יבדוק נוכחות מהנייד).

### בדיקות שכדאי להוסיף (הרחבת רשת הביטחון)
- **E2E מלא בענן** — כרגע יש תשתית Playwright ו-2 תרחישים; להרחיב לתרחישי הליבה (הרשמה ← קבלה ← סטודנט ← ציון ← יתרה) ולהריץ אוטומטית ב-CI.
- **בדיקות הרשאה שיטתיות** — טסט לכל endpoint שמוודא ש"בלי הרשאה = 403". סבב ה-QA הראה שכאן מסתתרים הבאגים המסוכנים.
- **בדיקת מיגרציות מול המסד** — סקריפט שמוודא ש-`types/database.ts` לא סוטה מהסכימה האמיתית (הסחיפה הזו כבר נתפסה שלוש פעמים).
- **בדיקות עומס קלות** — לוודא שהרשימות הגדולות מהירות עם 1,000+ רשומות.

### חוב טכני לניקוי (לא דחוף)
- ולידציית Zod ב-endpoints הישנים (החדשים כבר מכוסים).
- ניקוי הרשאות "מתות" משאריות seed 002 שתועדו.
- החלפת מטמון ההרשאות שבזיכרון (לא אפקטיבי ב-serverless) בפתרון עקבי.

## 7. שורה תחתונה

יומיים של **התבגרות**, לא רק גדילה. כל המודולים הושלמו (18/18), אבל
הסיפור האמיתי הוא שהמערכת עברה סבב אבטחה שמצא ותיקן פרצה קריטית (זיוף
superadmin) ועוד ~14 ליקויים — בדיוק כשמערכת עוברת מ"פרויקט" ל"מוצר".
האימות העצמאי שלי (417/417 בדיקות, טיפוסים נקיים) מאשר שהכל יציב.

**ההתקדמות: ~86%.** מה שנשאר לגמור-פונקציונלית הוא בעיקר שתי תשתיות
(התראות, קבצים) והכנה להשקה — כ-1.5–2.5 שבועות. לגבי ה-UI: הכיוון הנכון
הוא בדיוק מה שאמרת — לא רפורמה עכשיו, אלא שבוע ממוקד של **אחידות** (5
המשימות בסעיף 5), ולהשאיר את השדרוג הגדול לשלב מאוחר יותר כשהפונקציונליות
כבר סגורה.
