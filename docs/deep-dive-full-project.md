# CampusSystem — סקירה טכנית מקיפה של כל הפרויקט

**תאריך:** 23.07.2026
**היקף:** כל המערכת מהיסוד — ארכיטקטורה, כל המודולים, והאתגרים ההנדסיים
הקשים, עם דוגמאות קוד מדויקות מהפרויקט עצמו.

> זה לא דוח יומי. זו סקירה של המערכת כולה — מה היא, איך היא בנויה, ואיפה
> היו הרגעים הקשים באמת. כל דוגמת קוד כאן נלקחה מהקובץ האמיתי (הנתיב
> מצוין), לא הומצאה.

## חלק 1 — מה זו המערכת, בגדול

CampusSystem היא מערכת ניהול (CRM + ERP) למוסד חינוכי תורני. היא מלווה
אדם לאורך כל מסע החיים במוסד:

```
מתעניין → ליד → מועמד → תלמיד → בוגר
```

ומסביב למסע הזה — 20 מודולים שמנהלים כל היבט: כספים, פנימייה, הסעדה,
מרפאה, פסיכולוג, בקרת איכות, כוח אדם, חברותא, יהדות, ביטחון, תחזוקה,
נותני חסות, מסמכים, דוחות, ועוד. בנוסף — **פורטל תלמידים** נפרד שבו
התלמיד רואה את הנתונים שלו.

**הסטאק:** Next.js 14 (React) + TypeScript בחזית ובשרת · Supabase
(PostgreSQL) כמסד · Vercel לאירוח · JWT ל-authentication. שלוש שפות מלאות
(עברית, רוסית, אנגלית).

**המספרים:** ~95,000 שורות קוד אפליקציה · ~10,000 שורות SQL (66 מיגרציות)
· 599 בדיקות אוטומטיות · 380+ קומיטים.

## חלק 2 — הארכיטקטורה, שכבה-שכבה

המערכת בנויה בשכבות נקיות. כל בקשה עוברת דרכן בסדר קבוע:

```
דפדפן → middleware (שומר בשער) → route handler (API)
   → בדיקת הרשאה → lib/ (לוגיקה) → Supabase (מסד) → בחזרה
```

1. **`middleware.ts`** — כל בקשה נעצרת כאן קודם. בודק שיש session תקין,
   ושהמשתמש רשאי לגשת למודול. בלי זה → redirect ל-login או 401.
2. **Route handlers** (`app/api/**`) — כ-110 נקודות קצה. כל אחת בודקת
   הרשאה ספציפית לפעולה (לא רק למודול).
3. **שכבת הלוגיקה** (`lib/**`) — כאן החוכמה: הרשאות, מנוע תהליכים, כספים,
   נוכחות. **מופרדת בכוונה מהמסך** — ולכן אפשר לבדוק אותה ב-599 בדיקות
   מהירות בלי מסד.
4. **המסד** (`supabase/`) — לא רק טבלאות. חלק מהלוגיקה הקריטית חיה
   **בתוך** PostgreSQL כפונקציות אטומיות (נסביר למה זה קשה בחלק 3).

**עיקרון מפתח שחוזר בכל המערכת:** הפרדת לוגיקה טהורה מגישה למסד. דוגמה
אמיתית מחישוב תפוסת פנימייה (`lib/dormitory/occupancy.ts`) — פונקציה
בלי שום גישה למסד, ולכן קלה לבדיקה:

```ts
/** האם שיבוץ פעיל בתאריך נתון (טווח פתוח = ללא תאריך סיום). */
export function isActiveOn(a: Assignment, dateISO: string): boolean {
  return (
    a.status === 'active' &&
    a.assigned_from <= dateISO &&
    (a.assigned_to === null || a.assigned_to >= dateISO)
  )
}
```

## חלק 3 — האתגר הקשה ביותר: מנוע התהליכים

זה היה הרגע ההנדסי הקשה בפרויקט כולו, ושווה להבין למה.

### הבעיה

"תהליך גיוס" הוא שרשרת שלבים: קשר → מסמכים → אירוע → החלטה. כשמשלימים
שלב, המערכת צריכה לעשות **15+ פעולות ברצף**: לסמן את השלב כגמור, לבדוק
אם הוא סוגר את התהליך, להפעיל את השלבים הבאים, ליצור להם משימות, לדלג על
שלבים בלתי-נגישים, ואולי להמיר ליד למועמד.

בגרסה הראשונה כל זה נעשה ב-TypeScript, פעולה אחרי פעולה, כל אחת קריאה
נפרדת למסד. **הבעיה:** אם השרת קרס באמצע (פעולה 7 מתוך 15), נשאר **מצב
חלקי** — תהליך חצי-סגור, משימות יתומות, שלבים תקועים. במערכת שמנהלת
קבלת תלמידים, זה אסון: אין דרך לדעת מה קרה ומה לא.

### הפתרון — טרנזקציה אטומית ב-PostgreSQL

כל 15 הפעולות הועברו לתוך **פונקציה אחת ב-PostgreSQL** שרצה כטרנזקציה
יחידה: או שהכול מצליח, או ששום דבר לא קורה. אין אמצע. הנה תיאור המורכבות
מתוך הכותרת האמיתית של הקובץ (`supabase/migrations/20260703120000_complete_stage_rpc.sql`):

```
-- Атомарное завершение подэтапа + продвижение процесса — одной транзакцией.
-- ... самую сложную и рискованную функцию движка ...
-- 15+ последовательных операций, 2 крупные ветки (closes_process /
-- обычный поток), after_one/after_all, каскадный skip недостижимых подэтапов.
```

### מה שהופך את זה למקצועי

ההמרה לא "תיקנה" את ההתנהגות — היא שמרה אותה **אחד-לאחד**, כולל דקויות
עדינות שתועדו קודם. למשל, מהקובץ עצמו:

```
-- application_date НЕ трогается ни в одной из веток (в отличие от
--   close_process_early) — оставлено как в оригинале.
-- Skip недостижимых (5b) — best-effort (savepoint): сбой skip не
--   откатывает уже сделанные активации.
```

לפני ההמרה נכתבו **שלושה מסמכים**: ניתוח סיכונים, מסמך הכנה עם פירוק כל
הענפים, ומסמך baseline של ההתנהגות הקיימת. זו לא עבודה של מתחיל — זו
גישה של מהנדס שיודע שהחלק הזה מסוכן ומתייחס אליו בהתאם.

## חלק 4 — האתגר השני: מודל ההרשאות ההיררכי

זה החלק שהכי ייחודי למערכת שלך, כי הוא מגלם את המבנה הארגוני: מנהל →
ענפים → מזכירות → מורים → תלמידים, עם האצלת סמכויות.

### שלוש שכבות של "מי רואה מה"

הבסיס — לכל הרשאה יש **טווח** (`lib/permissions/scope.ts`):

```ts
export type Scope = 'all' | 'department' | 'own'
// all = כל המוסד · department = היחידה שלי · own = רק שלי
```

וכשלאדם כמה תפקידים עם טווחים שונים — **הגבוה מנצח** (`reduceScopes`).

### האצלה אישית — "המנהל מעניק"

מעל התפקידים יש שכבת האצלה אישית (`person_privileges`). המנהל יכול
להעניק סמכות לאדם ספציפי — אבל עם **תקרות קשיחות** (מהקוד האמיתי,
`app/api/education/units/[unitId]/members/[personId]/privileges/route.ts`):

```ts
if (!isHead) {
  for (const code of toGrant) {
    // נציג לא יכול להעניק מחדש את "זכות ההענקה" עצמה:
    if (code === 'delegate_privileges') return apiError('delegate_no_redelegate', 403)
    // נציג יכול להעניק רק הרשאה שיש לו בעצמו:
    const holds = await hasEducationPrivilege(session, code, { department_id: params.unitId })
    if (!holds) return apiError('delegate_cap', 403)
  }
}
```

התרגום לעברית: **אי אפשר להאציל סמכות שאין לך, ואי אפשר לשכפל מאצילים
בלי אישור הראש.** זה מודל שרואים במערכות ארגוניות רציניות.

### ירושת-עץ — הפער שנסגר לאחרונה

האתגר האחרון: כשמנהל ענף מקבל טווח "מחלקה", האם הוא רואה את תת-המחלקות
שתחתיו? במקור — לא (בדיקה שטוחה). זה תוקן ב-22.07 עם פונקציה שמטיילת
בעץ הארגוני (`lib/permissions/scope.ts`):

```ts
/**
 * Иерархический scope: расширяет ПРЯМО назначенные подразделения вниз по
 * дереву departments.parent_id — к каждому корню добавляются ВСЕ потомки.
 * Свойства (важно для безопасности): добавляются ТОЛЬКО потомки — ничего
 * «вбок» или «вверх». Циклобезопасно (visited-множество).
 */
export function expandDepartmentTree(rootIds: string[], edges: DepartmentEdge[]): string[] {
  const result = new Set<string>()
  const stack = [...rootIds]
  while (stack.length > 0) {
    const id = stack.pop()!
    if (result.has(id)) continue
    result.add(id)
    const children = childrenByParent.get(id)
    if (children) for (const c of children) if (!result.has(c)) stack.push(c)
  }
  return Array.from(result)
}
```

שים לב לדקויות האבטחה: **רק כלפי מטה** (צאצאים), אף פעם לא הצידה או
למעלה, ועם הגנה מפני לולאות אינסופיות. מנהל בכיר רואה אוטומטית את כל
הענף שתחתיו — בדיוק ההיררכיה שרצית.

## חלק 5 — האתגר השלישי: אבטחה (3 סבבים)

מערכת שמנהלת נתוני קטינים, כספים ותיקים רפואיים — האבטחה בה קריטית.
נערכו שלושה סבבים שמצאו ותיקנו פרצות אמיתיות.

### הפרצה הקריטית: זיוף מנהל-על

בקוד הישן, אם מפתח ה-JWT הסודי לא הוגדר, המערכת השתמשה בערך ברירת מחדל
**ידוע**. משמעות: מי שיודע את הערך יכול לזייף session של superadmin
ולהשתלט. התיקון (`lib/auth/config.ts`) — "fail-closed":

```ts
export function getJwtSecret(): string {
  const secret = process.env.JWT_SECRET
  if (!secret || secret.length < 32 || secret === DEFAULT_INSECURE_SECRET) {
    if (process.env.NODE_ENV === 'production') {
      throw new Error('JWT_SECRET is missing, too short, or insecure default...')
    }
    return DEFAULT_INSECURE_SECRET  // dev/test בלבד
  }
  return secret
}
```

בפרודקשן המערכת פשוט **מסרבת לעלות** בלי מפתח חזק. עדיף שתיפול מיד
מאשר שתרוץ עם דלת פתוחה.

### עוד פרצות שנסגרו

- **חשיפת PII** — כל מי שראה אנשים ראה גם דרכון וכתובת. נוצרה הרשאה
  נפרדת `persons.view_sensitive` שניתנת רק לקבוצה מצומצמת.
- **מירוץ תנאים בכספים (TOCTOU)** — שני אנשים שמאשרים תשלום בו-זמנית
  יכלו ליצור כפילות. אישור התשלום הפך לכתיבה אטומית מותנית (409 לשני).
- **בידוד פורטל התלמידים** — טבלת credentials נפרדת; login של תלמיד מקבל
  **אפס** הרשאות צוות, ולהפך.

## חלק 6 — האתגר הרביעי: סחיפת מיגרציות

זה אתגר "שקט" אבל נתפס **ארבע פעמים** במהלך הפרויקט. הבעיה: המיגרציות
מורצות ידנית דרך Supabase Dashboard, וקרה שמיגרציה מסוימת מעולם לא רצה
על המסד החי — הקוד ציפה לטבלה/הרשאה שלא הייתה שם, ופתאום כולם קיבלו 403.

**שני פתרונות אומצו:**

1. **מיגרציות "זורעות את עצמן"** — כל מיגרציה חדשה מוסיפה את ההרשאות
   שלה באופן אידמפוטנטי (`ON CONFLICT DO NOTHING`), במקום לסמוך על מה
   שהיה אמור לרוץ קודם.
2. **כלי אימות** (חדש, `scripts/verify-migrations.mjs`) — סורק את כל
   המיגרציות ומשווה מול הסכימה החיה, ומדווח על כל פער. כך הסחיפה לא
   יכולה עוד להתחבא.

## חלק 7 — מוקשי Supabase שלמדנו לזהות

שלושה "מוקשים" שגרמו לבאגים שקטים, ותועדו כחוקי פרויקט מחייבים (ב-CLAUDE.md):

1. **ה-builder אינו Promise** — `await sb.from(...).insert(...).catch(...)`
   שובר את הבנייה ב-Vercel. חייבים `const { error } = await ...`.
2. **שם FK לא משתנה בשינוי שם טבלה** — אחרי `RENAME`, embed עם שם ה-FK
   הישן מחזיר **null בשקט** במקום שגיאה. באג שקשה מאוד לאתר.
3. **`full_name` הוא שדה מחושב** (`GENERATED ALWAYS`) — אסור לכתוב אליו
   ב-INSERT/UPDATE, אחרת שגיאה.

הזיהוי של אלה — וההפיכה שלהם לחוקים כתובים — זה מה שמנע חזרה עליהם.

## חלק 8 — האתגר החמישי: תלת-לשוניות בלי לשבור

## חלק 10 — רשת הביטחון

מה שמאפשר לשנות מערכת של 95 אלף שורות בלי פחד:

- **599 בדיקות אוטומטיות** — רצות בשנייה, בלי מסד, על הלוגיקה המסוכנת

- **staging** — סביבת בדיקות נפרדת עם עותק סכימה.
- **Sentry** — ניטור שגיאות בזמן אמת (הותקן ונבדק לאחרונה).
- **תלת-לשוניות נאכפת** — ה-CI מוודא שאף מפתח תרגום לא חסר באף שפה.

## חלק 11 — שורה תחתונה

CampusSystem היא לא "אפליקציה" — היא **מערכת ארגונית** של 20 מודולים,
95 אלף שורות, עם שלושה אתגרים הנדסיים אמיתיים שנפתרו נכון: מנוע תהליכים
אטומי, מודל הרשאות היררכי מלא, ואבטחה שעברה שלושה סבבים. מה שמייחד אותה
זה לא רק ההיקף אלא **האיכות** — כל חלק קשה טופל בזהירות (מסמכי הכנה,
baseline, בדיקות), ולא "בערך".
